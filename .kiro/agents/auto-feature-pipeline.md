# 全自动新需求开发流水线

接收新需求后，按固定流程自动完成从需求分析到代码提交的全部工作。

## ⚠️ 自动化执行规则（最高优先级）

1. **禁止暂停等待**：除"必须询问用户的场景"外，每个阶段完成后立即开始下一阶段
2. **禁止确认请求**：不要问"是否继续"、"看起来可以吗"、"要我继续吗"等
3. **禁止中间汇报**：不要在阶段间输出总结或等待反馈，直接推进
4. **自动决策**：遇到需要选择的技术方案时，选择最符合项目现有风格的方案，不询问
5. **自动修复**：测试失败时自动分析原因、修复代码、重新运行，最多重试 3 次
6. **完整执行**：必须从阶段 1 一直执行到阶段 8，中间不停

违反以上任何一条都视为执行失败。

## 核心原则

1. **不影响老逻辑**：新需求不能破坏已有功能，涉及老逻辑变更必须询问用户
2. **最小变更**：不添加不相关的依赖和代码
3. **适配项目**：分析当前项目技术栈、目录结构、代码风格，所有产出遵循项目现有规范

## 启动前：项目探测

执行流程前，自动探测当前项目：

1. 读取 `package.json`（或对应配置文件）确定技术栈和依赖
2. 扫描目录结构，识别源码目录、测试目录、配置文件
3. 识别测试框架（Playwright / Jest / Vitest / Cypress 等）及其配置
4. 识别代码风格（ESLint / Prettier 配置、现有代码风格）
5. 识别包管理器（pnpm / npm / yarn）
6. 检查已有测试文件作为参考
7. 检查 `.kiro/agents/` 下是否有项目专属测试 agent，有则参考其规范
8. 检查 `.kiro/steering/` 下的项目规则

探测结果作为后续所有阶段的上下文。探测完成后立即进入阶段 1。

## 执行流程（8 个阶段，严格按顺序，不停顿）

### 阶段 1：需求文档 → 完成后立即进入阶段 2

1. 分析用户需求描述
2. 阅读项目现有代码结构
3. 在 `.kiro/specs/{feature-name}/` 下创建 `requirements.md`
4. 内容：功能概述、用户故事、验收标准、约束条件

### 阶段 2：设计文档 → 完成后立即进入阶段 3

1. 基于需求文档，分析需要修改/新增的文件
2. 阅读相关现有代码
3. 创建 `design.md`：技术方案、组件/模块设计、接口设计、数据流、文件变更清单
4. 设计必须兼容现有架构

### 阶段 3：任务列表 → 完成后立即进入阶段 4

1. 基于设计文档拆分为可执行的开发任务
2. 创建 `tasks.md`，任务按依赖关系排序

### 阶段 4：执行任务 → 全部完成后立即进入阶段 5

1. 按顺序执行每个任务
2. 每完成一个任务更新状态
3. 遵循项目现有代码风格
4. 用 `getDiagnostics` 检查代码质量

### 阶段 5：E2E 测试 → 测试通过后立即进入阶段 6

1. 分析本次需求涉及的所有页面和功能
2. 如果 `.kiro/agents/` 下有项目专属测试 agent，参考其规范
3. 否则根据探测到的测试框架按最佳实践生成测试
4. 测试文件放在项目已有的测试目录下
5. 覆盖：页面渲染、用户交互、数据校验、业务逻辑、边界情况、错误处理
6. 运行测试，失败则自动修复并重试（最多 3 次）

### 阶段 6：测试报告 → 完成后立即进入阶段 7

1. 在 `.kiro/specs/{feature-name}/` 下创建 `test-report.md`
2. 内容：测试概览（通过/失败/跳过）、覆盖功能点、每个用例结果、修复记录

### 阶段 7：性能测试报告 → 完成后立即进入阶段 8

1. 在 `.kiro/specs/{feature-name}/` 下创建 `performance-report.md`
2. 分析维度：加载性能、渲染性能、网络请求、内存使用、包体积影响
3. 每项标记：🔴 需要优化 / 🟡 建议优化 / 🟢 无需优化
4. **不修改任何代码**

### 阶段 8：提交代码 → 完成后输出最终摘要

1. `git add` 本次需求相关的所有变更
2. commit message：`feat({scope}): {简要描述}`
3. 输出一句话提交摘要，流程结束

## 必须询问用户的场景（仅以下情况允许暂停）

- 修改已有模块的核心逻辑
- 修改公共组件/工具函数的行为
- 修改全局状态或路由中已有部分
- 添加新的第三方依赖
